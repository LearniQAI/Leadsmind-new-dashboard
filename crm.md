# CRM & Sales Module — Independent Workflow Audit

## Summary
- Submodules audited: 13
- Workflows traced end-to-end: 24
- Confirmed fully working: 11
- Broken or partially broken: 20
- New/undocumented findings: 8

Scope note: this is a fresh static/code workflow audit of the current repository state. I did not execute live third-party integrations such as Google Places or email delivery, and I did not reuse prior audit conclusions.

## Submodule-by-Submodule Findings

### Contacts Management
Doc claim: Create, view, edit, delete contacts with full profile management.

Verified reality: Partially working. The real implementation is the top-level `/contacts` route, not `/crm`. `src/app/contacts/page.tsx` loads workspace-scoped contacts and tags, then renders `ContactsClient`. `src/components/crm/ContactForm.tsx:52-88` calls `createContact`/`updateContact`, and `src/app/actions/contacts.ts:147-190` persists through `ContactService` and `ContactRepository`. Repository writes are workspace-scoped in `src/modules/crm/repository/ContactRepository.ts:73-100`.

Workflow trace result: manual create/update/delete persists to `contacts` and revalidates `/contacts`. However `createContact` only checks `getCurrentWorkspaceId` and does not call `requireWorkspaceAccess` (`src/app/actions/contacts.ts:147-164`), unlike update/delete (`src/app/actions/contacts.ts:170-183`). If workspace selection can be manipulated, create is weaker than other mutations. Bulk delete reports success even if some deletes fail: `ContactsClient` counts successes but always shows `toast.success` with the count (`src/app/contacts/ContactsClient.tsx:82-96`), so partial failures are easy to miss.

### Contact Detail View
Doc claim: Dedicated contact page with activity feed, notes, tasks, tags.

Verified reality: Partially working. `/contacts/[id]` exists and uses server actions `getContact`, `getContactActivities`, `getContactNotes`, and `getContactTasks` from `src/app/actions/contacts.ts:46-83`. Notes and tasks are rendered in `ProfileContent` via `NotesManager` and `TasksManager` (`src/components/crm/ProfileContent.tsx:64-68`).

Workflow trace result: viewing a contact is workspace-scoped by repository filters (`ContactRepository.ts:9-45`). Adding notes/tasks persists (`ContactRepository.ts:240-266`). Broken detail: task delete succeeds in the UI without checking the server result; `TasksManager` always shows `Task deleted` after `deleteTask(deleteId)` (`src/components/crm/TasksManager.tsx:37-41`). Task completion similarly calls `toggleTaskStatus` without surfacing failures (`src/components/crm/TasksManager.tsx:84-87`).

## Cross-Module Workflow Traces

Lead Finder search -> CRM contact: `/lead-finder` calls `searchGooglePlaces`, saves search/results, results UI calls `addLeadsToCRM`, and that inserts contacts plus marks result rows `added_to_crm` (`src/app/actions/lead-finder.ts:10-224`). Outcome: contact creation works, but no opportunity/activity is created.

Lead workspace -> pipeline opportunity: `LeadCRMConnector` calls `pushLeadToPipeline` with literal `'default-pipeline-id'` and `'default-stage-id'` (`src/components/lead-finder/LeadCRMConnector.tsx:10-15`). Outcome: broken for real workspaces unless those fake IDs happen to exist.

Contact -> deal: `/pipelines` loads contacts and stages (`src/app/pipelines/page.tsx:43-50`); `OpportunityModal` lets the user select contact and stage (`src/app/pipelines/components/OpportunityModal.tsx:216-247`); `createOpportunity` persists to `opportunities` (`src/app/actions/pipelines.ts:68-105`). Outcome: works.

Deal -> contact detail: opportunity card links to `/contacts/{id}` (`src/app/pipelines/components/OpportunityCard.tsx:46-53`). Outcome: works for deals with contacts.

Form submission -> contact -> activity -> submission inbox: public submit creates/updates `contacts`, inserts `contact_activities`, inserts `form_submissions`, dispatches webhooks and automation trigger (`src/app/api/public/forms/[id]/submit/route.ts:262-458`). Outcome: partially works; contact sync/activity can silently fail or become inconsistent with submission insert.

Quote -> invoice: `QuoteMasterLedger` exposes Convert to Invoice only for accepted quotes (`src/components/quotes/QuoteMasterLedger.tsx:229-232`), calling `convertQuoteToInvoice` (`src/app/actions/quotes.ts:7-75`). Outcome: functional but not workspace-scoped and not atomic.

Campaign -> contacts: builder sends tags/emails to `updateCampaign` (`src/app/campaigns/[id]/builder/EmailBuilderClient.tsx:209-227`); direct emails are sent immediately (`src/app/actions/marketing.ts:383-396`); tag segments are counted but not queued in the traced path (`src/app/actions/marketing.ts:370-379`). Outcome: partial/silent scheduling risk.

CRM dashboard/activity: `/crm` queries `crm_opportunities`, `crm_contacts`, and `crm_activities` (`src/app/actions/crm-workspace.ts:15-48`) while the rest of the active CRM uses `opportunities`, `contacts`, and `contact_activities`. Outcome: competing implementation; dashboard can omit real module activity/data.

## Broken / Partially Broken Workflows

1. **RESOLVED** (2026-07-16). Contacts create does not enforce membership with the same `requireWorkspaceAccess` used by update/delete. Severity: High, cross-tenant/security concern. Files/functions: `src/app/actions/contacts.ts:147-164`, `requireWorkspaceAccess` at `src/app/actions/contacts.ts:19-43`. Fix: `createContact` now calls `requireWorkspaceAccess()` and uses the returned, membership-verified `workspaceId` for the insert, matching `updateContact`/`deleteContact`. Audit note: `requireWorkspaceAccess` checks an authenticated user *and* `workspace_members` row; `getCurrentWorkspaceId` only reads the client-supplied `active_workspace_id` cookie with no auth/membership check at all (`src/lib/auth.ts:121-124`). A broader grep of `src/app/actions/contacts.ts` shows the same weaker `getCurrentWorkspaceId`-only pattern is still used by `getContact`, `getContactActivities`, `getContactNotes`, `getContactTasks`, `globalDeleteTag`, `globalRenameTag`, `createRegistryTag`, `checkDuplicateContact`, `addTag`, `bulkAddTag`, `bulkRemoveTag`, `createNote`, `createTask`, `toggleTaskStatus`, and `searchContacts`. Those are out of scope for this pass (only the contacts-table create/update/delete asymmetry was in scope) but represent the same class of issue on tags/notes/tasks mutations and should be tracked as a follow-up.

2. **RESOLVED** (2026-07-16). Bulk contact delete can report success even when some records fail. Severity: Medium, silent failure. Files/functions: `src/app/contacts/ContactsClient.tsx:82-96`. Fix: `handleBulkDelete` now tracks failed IDs; all-success keeps the original success toast, partial failure shows an error toast with counts and re-selects only the failed IDs for retry, and all-failure shows a failure toast instead of a zero-count success.

3. **RESOLVED** (2026-07-16). Contact task delete/toggle can appear successful without checking the action result. Severity: Medium, silent failure. Files/functions: `src/components/crm/TasksManager.tsx:37-41`, `src/components/crm/TasksManager.tsx:84-87`. Fix: both handlers now await the server action and branch on `res.success`, showing an error toast on failure instead of an unconditional success toast/toggle. Root cause fixed at the data layer too: `ContactRepository.updateTaskStatus`/`deleteTask` previously issued `.update()`/`.delete()` with no `.select()`, so a mismatched `id`/`workspace_id` matched zero rows but still returned no error (silent no-op). Both now `.select('id')` and throw `NotFoundError` when zero rows are affected, so the action layer surfaces a real failure.

4. **RESOLVED** (2026-07-16). Task assignment lacks due-date input despite displaying due dates. Severity: Low/Medium, partial feature. Files/functions: `src/components/crm/TasksManager.tsx:24-28`, `src/app/actions/contacts.ts:254-263`. Fix: added a due-date field to the task creation UI using the existing `PremiumDatePicker` (`variant="light"`, the same component/pattern already used by the kanban Create Task modal), threaded `dueDate` through `createTask` (action → `ContactService.createTask` → `ContactRepository.createTask`) into the `contact_tasks.due_date` column. The column already existed in the initial schema (`supabase/migrations/20240101000002_phase2_initial_schema.sql:52`), so this was a wire-up fix, not a schema change.

10. **RESOLVED** (2026-07-16, architectural). Global activity feed reads `crm_activities` while contact/form workflows write `contact_activities`. Severity: High, broken cross-submodule history. Files/functions: `src/lib/crm/UnifiedActivityEngine.ts:36-48`, `src/app/api/public/forms/[id]/submit/route.ts:378-395`, `src/modules/crm/repository/ContactRepository.ts:18-27`. **Investigation finding: neither table is legacy — both are live and actively written by disjoint code paths, so this is a genuine two-source-feed situation, not a canonical-vs-dead one.** `crm_activities` was created later (`supabase/migrations/20240101000121_phase78_unified_crm.sql`, commented "Unified Activities") alongside a parallel `crm_contacts`/`crm_companies`/`crm_opportunities` schema, but unlike those tables (which finding #20 shows are only read by the old `/crm` dashboard and are effectively dead weight), `crm_activities` is written from many live, actively-used code paths — automation workflows (`WorkflowEngine.ts`), task creation (`task-workspace.ts`), follow-ups (`FollowupManager.ts`), escalations (`EscalationHandler.ts`), WhatsApp/email messaging (`messaging.ts`, `CRMActionHandler.ts`) — and its `entity_id` for `entity_type: 'contact'` rows was confirmed (via `WorkflowEngine.ts`) to reference the real `contacts.id`, not the parallel `crm_contacts.id`. `contact_activities` is written by the contact-detail page (create/notes/tags) and public form submissions. One code path (`WorkflowEngine.ts`'s bounce-failover handler) already writes to *both* tables for related events, which is itself evidence this split is an organic inconsistency rather than an intentional two-tier design — but since both are genuinely live, **the fix merges rather than migrates**: `UnifiedActivityEngine.getGlobalActivity` now queries both tables, normalizes each into a common shape, and returns one correctly-sorted merged timeline (`src/lib/crm/UnifiedActivityEngine.ts`). `UnifiedActivityFeed.tsx` and the full `/crm/activity` page now link the entity badge to `/contacts/{entity_id}` when `entity_type === 'contact'`; other entity types (`company`/`opportunity`/`form`/`lead`) are left unlinked since no confirmed working detail route exists for them (see finding #17 — `/opportunities/[id]` doesn't exist), so linking them would just add new dead links. **Note for future work on this feed: do not duplicate writes into both tables as a "quick fix" — the two-table split works today only because the merge query treats them as one logical stream at read time; if a third activity source appears, extend the merge, don't add a third write target.**

14. **RESOLVED, full atomic RPC, verified live** (2026-07-19). Quote-to-invoice conversion is not atomic; invoice can be created without quote marked converted. Severity: High, data integrity/data loss risk. Files/functions: `src/app/actions/quotes.ts:31-66`. Line items could not be carried over during conversion because quotes have no line-item storage in the live schema at all (the `invoice_items` table has a `quote_id` column for this purpose but nothing in the app ever writes to it) — flagging this as a separate, pre-existing data-model gap, out of scope to fix here. **Verified live**: converting a real accepted test quote produced exactly one invoice and correctly updated the quote (`status: 'converted'`, `converted_invoice_id` set); calling conversion again on the same quote returned `already_converted: true` with the same `invoice_id` and created no second invoice; calling with a mismatched `workspace_id` returned `Quote not found` rather than converting. Test data cleaned up after.

## Part B — `pipelines.ts` Weak-Tier Fix (Priority 4 of the platform-wide security effort, 2026-07-17)

**#21 — RESOLVED.** The project-wide detection pass flagged in #13b/A1's closing note (33 files with the "unprotected `workspaceId`-taking action" pattern) eventually reached `pipelines.ts` as part of a platform-wide Priority 0–4 remediation effort tracked separately in `security-remediation.md` at the project root. `pipelines.ts` is conceptually CRM-scoped (pipeline/stage/opportunity management, the same "deals" surface #2/#5 above already cover) even though it was fixed as part of that platform-wide batch rather than a dedicated CRM session — noting it here so this file's audit trail stays complete.

**`updateStageOrder`'s RPC investigated, not assumed safe:** this function calls `update_stage_positions(p_workspace_id, p_stage_ids, p_positions)`, a `SECURITY DEFINER` Postgres function. **Bug #22b — Sales Pipeline Summary per-stage deal counts were structurally broken via a PostgREST count-embed misread.** Severity: High, the sharper of the two — this bug does not throw and does not look obviously broken. `src/lib/analytics.ts`'s `fetchDashboardMetrics` (powers the pipeline funnel row) queried `pipeline_stages` with `.select('id, name, position, opportunities(count)')` and computed each stage's deal count as `Array.isArray(s.opportunities) ? s.opportunities.length : 0`. **Context:** a fourth pass on this module, prompted by two fresh reports: adding a deal via `/pipelines` produces a success toast but the deal never appears on any stage card (a distinct, sharper complaint than Part C's "doesn't reflect on `/dashboard`" — this one is about the `/pipelines` board itself, never previously audited end-to-end for its own rendering), plus a fifth recurrence of the "R 0.00 reads as R O.OO" currency bug, this time on this same page. ### Part A — "Deal added but invisible": root cause was a live PostgREST embedding-ambiguity error, not a client-refresh bug

Static tracing of `PipelinesClient.tsx`/`OpportunityModal.tsx`/`pipelines.ts` initially suggested everything *should* work — `createOpportunity` writes a correct, correctly-scoped row, calls `revalidatePath('/pipelines')`, and the modal calls `router.refresh()` inside `startTransition()` on success, against a `force-dynamic` page. Rather than trust that theoretical trace, the write+read path was tested live against the real linked Supabase project (throwaway workspace/pipeline/stage, cleaned up after) using the exact insert and query shapes the app's own code uses — and the read failed outright:

```
code: 'PGRST201'
message: "Could not embed because more than one relationship was found for 'opportunities' and 'contacts'"
hint: "Try changing 'contacts' to one of the following: 'contacts!opportunities_buyer_id_fkey', 'contacts!opportunities_contact_id_fkey', 'contacts!opportunities_seller_id_fkey'"
```

**Root cause, confirmed not guessed:** `20240101000210_sprint11_real_estate_pipeline.sql` added `buyer_id`/`seller_id` foreign keys from `opportunities` to `contacts`, alongside the pre-existing `contact_id` FK — giving `opportunities` **three** foreign keys into `contacts`. `getPipelineOpportunities`'s unqualified `.select('*, contact:contacts(*)')` (`pipelines.ts`) became genuinely ambiguous to PostgREST the moment that migration ran, and has been failing on **every single call, for every workspace, unconditionally** — not just after a create. `page.tsx`'s `(opportunitiesResult as any).success ? ... : []` fallback silently swallowed the error into an empty array, so the board has shown "No deals yet" regardless of real data since that migration, full stop — this was never actually about a freshly-created deal specifically, that was just the most noticeable symptom.

This is a different bug from Part C's #22b (`opportunities(count)` aggregate-array misread in `analytics.ts`) — confirmed distinct by reproducing #22b's exact pattern side-by-side with this one in the same live test and showing they're structurally different PostgREST failure modes (one silently returns wrong data, this one refuses to run at all) — not a duplicate finding.

**Fix:** qualified the embed with the actual FK constraint name the app's own `contact_id` field uses: `contacts!opportunities_contact_id_fkey(*)`. Same ambiguity, same fix, also found and fixed in `src/app/dashboard/page.tsx`'s "High-Value Opportunities" query (`contacts(id, first_name, last_name)`, Part C's own `topOpportunities` — Part C had certified this query "working as designed" at the time, before the real-estate migration's `buyer_id`/`seller_id` columns existed or before this specific ambiguity was tested live; not a contradiction of that finding, an update to it — flagging directly rather than leaving Part C's claim uncorrected).

**Separately, also fixed (defense in depth, not the primary cause):** `PipelinesClient.tsx` had no direct code path connecting "deal created" to "board shows it" at all — only drag-and-drop ever updated the board's `useOptimistic` state; a fresh create relied entirely on `router.refresh()` indirectly cascading a new prop down through a re-render. Replaced with a real `useState` the create/update/delete flow (`OpportunityModal`'s new `onSaved` callback) updates directly and synchronously the instant the server action resolves, with `router.refresh()` kept as a consistency backstop rather than the sole mechanism. This wasn't the reason deals were invisible (the embed error was), but it's a genuinely more robust pattern than depending on an indirect prop cascade, and is now the same direct-update mechanism used for both mutation paths on this board instead of two different ones.

**Live-verified**, 10/10 checks against the real database: insert succeeds with correct `workspace_id`/`stage_id`; the fixed read query succeeds with no error and returns the new row; the board's exact `opp.stage_id === stage.id` client-side filter correctly includes it and doesn't leak into other stages; the count-embed bug pattern (#22b) was independently reproduced to confirm it's a real, distinct thing and not this same bug under another name.

### Part B — Fifth "R 0.00 → R O.OO" recurrence: the font-feature-settings fix has never once worked, on any of the five instances

Per the task's explicit instruction not to apply a sixth local patch: rather than re-guess at a font/weight tweak, the actual DM Sans font file this app loads (`fonts.googleapis.com` → `fonts.gstatic.com`, the real v17 file, confirmed by its own embedded copyright string) was downloaded and its OpenType GSUB feature table inspected directly (decompressed WOFF2 → TTF, parsed with `opentype.js`). Result: **9 features present (`calt, ccmp, dnom, frac, liga, locl, numr`) — no `zero` feature at all.**

This is the real, evidence-based root cause of all five "fixes" failing identically: `font-variant-numeric: slashed-zero` and `font-feature-settings: '"zero" 1'` both work by requesting the font's `zero` OpenType feature — on a font that doesn't implement it, the browser silently ignores the request, every time, with no error and no fallback. Every prior fix (Pipelines, Affiliates, Dashboard, Invoices) used exactly this mechanism, so every prior fix was inert from the moment it was written, regardless of how many places it got copied to. This was previously assumed to be a rendering/ambiguity issue about the glyph shape at small bold sizes — it's actually simpler and more mechanical than that: the requested fix mechanism was never capable of doing anything on this typeface.

**Fix, font-independent this time:** `CurrencyValue.tsx` (`src/components/dashboard-ui`, the shared component already established as this project's single source of truth for money formatting, per its own doc comment) now appends a Unicode combining long solidus overlay (U+0338) after every literal `"0"` character in the formatted string — a real character-composition slash through the zero, rendered by the text layout engine directly rather than requested from the font's feature table, so it works regardless of what font is active, now or in the future. Exported as `slashZeros()` for the two count displays on this page that aren't currency (Active Deals / Weighted Won) but had the same defunct fix applied to them too.

**Confirmed this page wasn't even using the shared component consistently** (the second half of why this kept recurring here specifically): `OpportunityCard.tsx` already used `CurrencyValue` — but `CurrencyValue` itself only had the inert font-feature-settings fix baked in, so using the "correct" shared component didn't help until the component's own root mechanism was fixed. `PipelineStats.tsx` and `KanbanColumn.tsx` never used `CurrencyValue` at all — each hand-rolled its own `R {value.toLocaleString(...)}` string plus its own local, also-inert `NUMERIC_STYLE` object, duplicating both the formatting logic and the non-working fix. All three call sites on this page now go through the one shared component/utility instead of three separate hand-rolled copies.

**Verified via direct character-level inspection** (not a visual glance, since no browser tool is available in this environment): confirmed programmatically that `slashZeros("0,00")` (note: `en-ZA` locale renders the decimal separator as a comma, not a period — a pre-existing, unrelated formatting detail, not touched) produces the correct codepoint sequence — U+0030 (`0`) immediately followed by U+0338 (combining slash) at both zero positions, comma untouched. This proves the mechanism is applied correctly to the right characters; actual on-screen rendering quality of the combining mark (how visually prominent the slash reads) could not be confirmed without a browser and is flagged for the user's own visual confirmation once deployed, same limitation as every other UI-verification item in this document.
